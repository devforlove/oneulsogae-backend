# 사업계획서

## 미플(Meeple) — 직장인 신원검증 기반 소개팅·소셜 매칭 플랫폼

> 정부지원사업 제출용(PSST 양식) · 본 문서의 기술/기능 내용은 실제 구현 코드 기준으로 작성되었으며,
> 시장규모·재무·팀 등 코드로 확인 불가한 항목은 **【작성 필요】**로 표시했습니다.

| 핵심 가치 | 회사 이메일 인증으로 검증된 직장인만 참여하는 신뢰 기반 매칭 |

## 1. 문제인식 (Problem)

### 1-1. 창업 배경 및 필요성

- **직장인의 만남 기회 부족**: 장시간 근로와 좁은 생활 반경으로 직장인의 자연스러운 만남 기회가 구조적으로 부족하다.
- **기존 소개팅앱의 신뢰 문제**: 익명·허위 프로필, 신원 미검증으로 인한 사기·불쾌 경험이 이용 이탈의 핵심 원인이다.
- **검증의 부재**: 직업·소속을 자기신고에 의존하는 구조는 "검증된 직장인"이라는 가치를 보장하지 못한다.

### 1-2. 목적 및 해결하고자 하는 문제

미플은 **회사 이메일 도메인 인증**을 온보딩 필수 절차로 두어, 신원이 검증된 직장인만 매칭 풀에 진입하게 한다.
이를 통해 "상대가 실제 직장인인가"라는 신뢰 비용을 플랫폼이 구조적으로 해결한다.

- 회사 이메일 인증번호 발송·검증 → 회사명 자동 매핑(미매핑 시 직접 입력)
- 인증 완료(ACTIVE) 사용자만 매칭 대상에 포함
- 온보딩 단계: ONBOARDING → EMAIL_VERIFICATION_PENDING → COMPANY_NOT_RESOLVED → ACTIVE

## 2. 실현가능성 (Solution)

### 2-1. 개발·구현 현황 (이미 동작하는 기능)

> 아래는 **실제 코드에 구현되어 동작하는 기능**으로, 본 사업의 실현가능성을 뒷받침하는 핵심 근거다.

**① 사용자 온보딩·프로필 (USER 도메인)**
- 회사 이메일 인증번호 발송·검증, 회사명 자동/수동 매핑으로 정식 가입(ACTIVE) 처리
- 프로필(직업·취미·종교·결혼·음주·흡연·체형·키 등) 등록/수정, 나이·성별·회사이메일은 변경 불가로 신뢰성 유지
- 지역 5권역 자동 분류(회사 이메일 도메인 기반)

**② 1:1 소개팅 매칭 (MATCH 도메인) — 완전 구현**
- 온보딩 직후/일일 배치로 반대 성별 1명 자동 소개(PROPOSED)
- 관심 표시 → 신청/수락 → 양측 수락 시 성사(MATCHED)
- 상태 흐름: PROPOSED → PARTIALLY_ACCEPTED → MATCHED / CLOSED
- 유효기간 24시간, 만료 자동 처리, 재소개 방지(조합 유니크 제약)

**③ 코인 경제 (COIN 도메인) — 완전 구현**
- 적립(출석/구매/환불)·차감(소개팅·미팅 신청/수락) 원장 + 잔액 물질화 모델
- 매칭 실패 시 수락자에게 신청 비용의 50% 환불
- 코인 상점 상품 목록, 잔액 조회

**④ 실시간 채팅 (CHAT 도메인) — 완전 구현**
- WebSocket(STOMP) 기반 실시간 메시지, REST 기반 채팅방 목록/상세(키셋 페이지네이션)
- 메시지 1건당 4쿼리로 최적화(발신자 검증·방 갱신·저장·미읽음 벌크 증분), N+1 제거
- 읽음 처리, 채팅방 나가기(본인 행만 소프트 삭제)

**⑤ 2:2 팀 결성 (MATCH-Team) — 결성까지 구현**
- 팀 초대 → 수락 → 결성(FORMED), 초대 철회·팀 해체
- 같은 성별만 팀 구성, 초대 가능 사용자 검색
- (페어링·성사 로직은 향후 구현 — 2-2 참조)

**⑥ 알람·팝업·인증 (지원 도메인)**
- 알람 목록(발신자 프로필 포함), 일일 보상 팝업(출석 코인 자동 적립), 환불 안내 팝업
- JWT 토큰 회전 재발급(쿠키 기반), 로그아웃, 토큰 재사용 감지

**⑦ 배치/스케줄 (SCHEDULER)**
- 일일 매칭 배치(성별·지역별 풀 관리, 재소개 방지)
- 매칭 만료 배치(24시간 경과 처리 + 환불 계산 + 팝업 생성)
- 사용자 활성 스냅샷 동기화

### 2-2. 향후 구체화 방안 (개발 로드맵)

| 항목 | 현황 | 계획 |
|------|------|------|
| 2:2 팀 페어링·성사 | 팀 결성·엔티티만 구현 | 팀×팀 매칭 알고리즘, 관심/수락, 성사 후 채팅방 생성 |
| 미팅(N:N) 매칭 | 코인 타입만 정의 | N:N 매칭 비즈니스 로직 구현 |
| 신청식 매칭 | 타입(REQUIRED) 정의 | 사용자 능동 신청 인터페이스/API |
| 차단·선호도 | 미구현 | 차단/이상형 필터 기반 추천 고도화 |
| 결제 연동 | 코인 적립 구조 존재 | PG 결제 연동(인앱 결제) |

### 2-3. 기술적 차별성 및 경쟁력

- **헥사고날 아키텍처(Ports & Adapters) + CQRS**: 도메인 핵심을 인프라에서 분리, 기능 확장·테스트 용이
- **신원검증 구조**: 회사 이메일 인증을 코어 온보딩에 내재화 → 경쟁사가 모방하기 어려운 진입장벽
- **성능 설계**: 채팅 메시지 4쿼리 제한, 키셋 페이지네이션, 인덱스 효율 고려한 쿼리 설계
- **테스트 체계**: 도메인 Kotest 유닛 + API E2E(Testcontainers)로 품질 검증

## 3. 성장전략 (Scale-up)

### 3-1. 비즈니스 모델 (코인 경제)

미플의 핵심 수익원은 **코인**이다. 매칭 행위(신청/수락)에 코인을 소모하고, 사용자는 코인을 구매하여 충전한다.

| 행위 | 코인 비용 | 비고 |
|------|----------|------|
| 소개팅 신청 | 32 코인 | DATING_INIT |
| 소개팅 수락 | 32 코인 | DATING_ACCEPT |
| 미팅 신청 | 40 코인 | MEETING_INIT(로직 구현 예정) |
| 미팅 수락 | 40 코인 | MEETING_ACCEPT(로직 구현 예정) |
| 매칭 실패 환불 | +16 코인 | 수락자에게 신청 비용 50% 환불 |
| 출석 보상 | 무료 적립 | 일일 1회 |

- **수익 구조**: 코인 구매(PURCHASE) → 매칭 소모. 무료 적립(출석)으로 리텐션 확보, 유료 충전으로 수익화.
- **단가/패키지 가격 정책**: 【작성 필요 — 코인 패키지별 판매 단가/원가/마진】

### 3-2. 시장 규모 및 진입 전략

- **목표 시장**: 신원검증을 중시하는 직장인 데이팅 시장
- **시장 규모(TAM/SAM/SOM)**: 【작성 필요 — 국내 데이팅앱 시장규모, 직장인 타깃 규모 통계】
- **진입 전략**: 【작성 필요 — 특정 회사/산업군 거점 확산, 초기 사용자 확보(CAC) 전략】
- **성과 목표**: 【작성 필요 — 가입자/MAU/유료전환율/매출 목표(연차별)】
